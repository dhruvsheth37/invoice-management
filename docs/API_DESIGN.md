# API Design

## 1. Conventions

- Base path: `/api/v1`
- Media type: `application/json`
- Dates: ISO 8601 calendar dates (`YYYY-MM-DD`)
- Timestamps: UTC ISO 8601
- Currency: ISO 4217 three-letter code
- Money: JSON decimal numbers; SQL `decimal(19,4)`
- Identifiers: GUIDs generated by the server
- Errors: RFC 9457-style `ProblemDetails`
- Successful responses: resource DTOs and standard HTTP status codes; no generic response envelope

## 2. Request context

| Header | Requirement | Purpose |
|---|---|---|
| `Authorization` | Production | JWT identity and trusted tenant claim |
| `X-Tenant-Id` | Development/test only | Local tenant resolution |
| `X-Development-User-Id` | Development only | Optional positive integer user ID; defaults to `1` |
| `traceparent` | Optional | W3C distributed trace context |
| `X-Correlation-ID` | Optional | Human-searchable operation correlation |
| `Idempotency-Key` | Required for mutating POSTs | Retry protection |
| `If-Match` | Required for lifecycle operations | Optimistic concurrency |

The server returns `X-Correlation-ID`, `traceparent` context through standard instrumentation, and `ETag` for invoice resources.

## 3. Endpoints

| Method | Route | Purpose |
|---|---|---|
| `POST` | `/api/v1/invoices` | Create a draft invoice |
| `POST` | `/api/v1/invoices/search` | Search tenant invoices using body filters |
| `GET` | `/api/v1/invoices/{invoiceId}` | Retrieve invoice details |
| `POST` | `/api/v1/invoices/{invoiceId}/issue` | Issue a draft |
| `POST` | `/api/v1/invoices/{invoiceId}/mark-paid` | Confirm an issued invoice as paid |
| `POST` | `/api/v1/invoices/{invoiceId}/void` | Void a draft or issued invoice |
| `GET` | `/api/v1/dashboard/invoice-summary` | Retrieve tenant summary |

There is no generic `PATCH /status`. Each lifecycle route expresses a business operation and validates allowed transitions.

No delete endpoint is implemented. Business records use `IsActive`; any future invoice deactivation operation must be limited to Draft invoices.

### 3.1 Why lifecycle operations use separate endpoints

`issue`, `mark-paid`, and `void` are separate business commands rather than three generic ways to update the `Status` field. Each command has a different source-state rule, request contract, validation, side effect, and audit meaning:

| Command | Allowed source state | Command-specific data | Important side effects |
|---|---|---|---|
| `issue` | Draft | Issue date and due date | Allocates the invoice number and captures the bill-to snapshot |
| `mark-paid` | Issued | Paid date and external reference | Records confirmation from the external payment/accounting process |
| `void` | Draft or Issued | Required reason | Cancels the invoice while retaining its financial history |

Separate endpoints provide:

- Strongly typed request models without unrelated optional fields.
- Clear OpenAPI documentation and client intent.
- Command-specific validation, authorization, idempotency scope, and audit records.
- Independent evolution when one lifecycle operation gains new fields or rules.
- Protection against clients assigning arbitrary statuses and bypassing domain behavior.

A combined endpoint such as `POST /invoices/{invoiceId}/transition` would require an action discriminator and a union of conditionally valid fields such as `issueDate`, `paidDate`, `reference`, and `reason`. It would move complexity into a controller switch, weaken compile-time request validation, and make authorization and API documentation action-dependent. If the lifecycle later grows into a large event-driven state machine, a discriminated command endpoint may become appropriate, but it should still dispatch to separate command handlers internally.

The three actions remain methods of the same `InvoicesController` and share the private response/ETag handling helper. This avoids implementation duplication without merging distinct public contracts.

## 4. Create draft

```http
POST /api/v1/invoices
Idempotency-Key: 2bd2a113-b966-455f-af22-2d4f30ef9690
X-Correlation-ID: portal-9f2d4a
```

```json
{
  "customerId": "73ecdfb4-fc0d-49f5-b256-b0b8450e0f3d",
  "customerLocationId": "ca37ad44-4c68-4d22-b92f-bde5d89a893c",
  "currencyCode": "USD",
  "dueDate": "2026-08-15",
  "notes": "Net 30",
  "lineItems": [
    {
      "description": "Freight handling",
      "quantity": 2.0000,
      "unitPrice": 125.0000,
      "taxRate": 0.180000
    }
  ]
}
```

Response: `201 Created`, `Location`, `ETag`, and the invoice detail representation.

Validation includes:

- Existing active customer and location in the current tenant.
- Location belongs to the selected customer.
- Supported uppercase currency code.
- At least one line item.
- Positive quantity, non-negative price, and tax rate between 0 and 1.
- Maximum field lengths and configured line-count limit.
- Totals calculated by the server.

## 5. List invoices

```http
POST /api/v1/invoices/search
Content-Type: application/json
```

```json
{
  "pageSize": 25,
  "status": "Issued",
  "customerId": "73ecdfb4-fc0d-49f5-b256-b0b8450e0f3d",
  "from": "2026-01-01",
  "to": "2026-12-31",
  "dueFrom": null,
  "dueTo": null,
  "invoiceNumber": null,
  "sort": "-createdUtc",
  "continuationToken": null,
  "includeTotalCount": false
}
```

Supported filters:

- `status`
- `customerId`
- `from` and `to` creation dates
- `dueFrom` and `dueTo`
- `invoiceNumber`

Rules:

- Default page size: 25.
- Maximum page size: 100.
- Default sort: `createdUtc` descending, then `id` descending for stability.
- Supported sorts are allow-listed.
- Pagination uses an opaque continuation token tied to the current filters and sort; offset pagination is not used.
- Exact total count is optional and executes only when `includeTotalCount` is `true`.
- Query uses `AsNoTracking` and DTO projection.
- Named `TenantFilter` and `ActiveFilter` filters automatically exclude other-tenant and inactive rows.
- `customerId` is an explicit search criterion. Billing location is not an authorization scope and is therefore not a global filter.

```json
{
  "items": [],
  "pageSize": 25,
  "totalCount": null,
  "continuationToken": null
}
```

## 6. Invoice details

```http
GET /api/v1/invoices/{invoiceId}
```

Response: `200 OK` with `ETag`. An absent, inactive, or other-tenant invoice returns the same `404 Not Found` response to avoid disclosing resource existence.

## 7. Issue invoice

```http
POST /api/v1/invoices/{invoiceId}/issue
Idempotency-Key: 98b21a12-fbc6-4051-9775-c39b6f1cb8a2
If-Match: "AAAAAAAAB9E="
```

Optional request:

```json
{
  "issueDate": "2026-07-16",
  "dueDate": "2026-08-15"
}
```

The operation validates Draft status, at least one active line, due date, and totals; atomically assigns the invoice number and records status history.

Issuance also copies the current customer legal name, tax number, and selected location address into immutable bill-to snapshot fields returned by subsequent detail requests.

Idempotency-Key: 3ab1e83d-e681-47eb-87be-48392f1663a7

This represents confirmation from an external payment/accounting process. A native payment and allocation model is a future enhancement.

## 9. Void invoice

```http
POST /api/v1/invoices/{invoiceId}/void
Idempotency-Key: 9c671415-4f97-4f0d-854f-adc87380f314
If-Match: "AAAAAAAAB9M="
```

```json
{
  "reason": "Created for the wrong customer location"
}
```

`reason` is required. Paid and already-void invoices cannot be voided in the assessment scope.

## 10. Dashboard summary

```http
GET /api/v1/dashboard/invoice-summary?asOf=2026-07-16
```

```json
{
  "asOf": "2026-07-16",
  "currencyBreakdown": [
    {
      "currencyCode": "USD",
      "draft": { "count": 2, "amount": 350.0000 },
      "issued": { "count": 4, "amount": 9200.0000 },
      "paid": { "count": 7, "amount": 14300.0000 },
      "void": { "count": 1, "amount": 200.0000 },
      "overdue": { "count": 1, "amount": 500.0000 }
    }
  ]
}
```

Amounts are never combined across currencies. `Overdue` is derived from Issued invoices with `DueDate < asOf`.

## 11. Error contract

```json
{
  "type": "https://invoice-management.example/problems/invalid-transition",
  "title": "Invoice transition is not allowed",
  "status": 409,
  "detail": "A paid invoice cannot be voided.",
  "instance": "/api/v1/invoices/8e8c.../void",
  "traceId": "d4f83e...",
  "correlationId": "portal-9f2d4a",
  "errorCode": "invoice.invalid_transition"
}
```

Expected statuses:

| Status | Use |
|---|---|
| `400` | Malformed input or invalid correlation header |
| `401` | Missing/invalid identity in production |
| `403` | Authenticated but not permitted |
| `404` | Resource unavailable in current tenant |
| `409` | Invalid transition, concurrency conflict, or idempotency conflict |
| `422` | Semantically invalid request fields |
| `429` | Rate limit exceeded |
| `500` | Unexpected error with no internal details exposed |

## 12. OpenAPI and evolution

- OpenAPI documents headers, status codes, idempotency, concurrency, and examples.
- Additive contract changes are preferred within v1.
- A new major route version is introduced only for breaking external contract changes.
- Internal module refactoring does not create an API version.
